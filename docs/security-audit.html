Security Findings

🔴
In-Memory Rate Limiter Resets on Server Restart Critical Open
checkRateLimit() uses a Map<string, {count, resetAt}> stored in process memory. On Vercel (serverless), every cold start resets all counters — meaning rate limits provide zero protection against distributed brute-force attacks. An attacker can simply hammer the sign-in endpoint from multiple invocations.
// current (src/lib/server/rate-limiter.ts) const store = new Map<string, {count: number; resetAt: number}>();
Fix: Replace with a Redis-backed limiter (Upstash Redis + @upstash/ratelimit). Upstash has a generous free tier. One import swap replaces the entire module.
// recommended import { Ratelimit } from "@upstash/ratelimit"; import { Redis } from "@upstash/redis"; const ratelimit = new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(10, "15 m"), });
🔴
Middleware Auth Check is Cookie-Presence Only Critical Open
The middleware checks request.cookies.has("authjs.session-token") — it does not verify the JWT signature. A crafted request with a forged or expired cookie named authjs.session-token would bypass the middleware gate and reach protected routes. The actual server actions do call auth() which validates the JWT, but this creates a misleading trust boundary and could expose UI to unauthenticated users.
Fix: Use the official auth() export in middleware (next-auth v5 supports this via export { auth as middleware } from "./lib/auth"). This properly validates the JWT on every edge request.
🟠
No CSRF Protection on Server Actions High Open
Next.js Server Actions include a built-in origin check for same-origin requests, but this protection only applies to form submissions. Direct fetch() calls with a valid session cookie from a malicious cross-origin site can invoke server actions. All balance/spin/admin mutations are affected.
Fix: Add a x-csrf-token header check in sensitive server actions, or enforce SameSite=Strict on session cookies. Next.js 15 also provides experimental.serverActionsBodySizeLimit and should have allowedOrigins configured in next.config.js.
🟠
Open Redirect via callbackUrl (Partially Fixed) High Fixed ✓
Previously the middleware passed the full absolute URL as callbackUrl, which allowed external redirects. The middleware now passes only the pathname. The sign-in page also sanitizes to require the path starts with / and not //. However, next-auth's own signIn() accepts arbitrary callbackUrl values from OAuth flows — ensure NEXTAUTH_URL is always set in production to restrict OAuth callback origins.
🟠
Password Reset Tokens Stored in Plain Text High Open
The verificationTokens table stores reset/verify tokens verbatim. If the database is breached, an attacker could directly use these tokens to reset any user's password. Tokens should be stored as SHA-256 hashes.
// recommended: store hash, compare hash const tokenHash = crypto.createHash("sha256").update(rawToken).digest("hex"); await db.insert(verificationTokens).values({ ..., token: tokenHash });
🟡
Admin Route Uses Layout-Level Guard Only Medium Fixed ✓
Each admin server action calls requireAdmin() which re-validates the session and role from the database on every request — this is correct. The layout-level redirect is defence-in-depth for the UI only. Architecture is sound.
🟡
No Content Security Policy (CSP) Header Medium Open
The app has no CSP headers configured. This means any XSS vulnerability (e.g. from user-supplied content in the future) could execute arbitrary scripts, exfiltrate session tokens, or redirect users.
Fix: Add CSP headers in next.config.js:
headers: async () => [{ source: "/(.*)", headers: [{ key: "Content-Security-Policy", value: "default-src 'self'; script-src 'self' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' https:; frame-ancestors 'none';", }], }]
🟡
No SQL Injection Protection Audit on Dynamic Queries Medium Info
The app uses Drizzle ORM for all queries — parameterized by default, making SQL injection highly unlikely. However, the admin search uses like(users.email, `%${query}%`) where query comes from user input. Drizzle parameterizes this correctly, but extremely long query strings could cause slow LIKE scans without an index.
Recommendation: Add a GIN/TRGM index on users.email and enforce a max 100-char limit on the search input.
🟢
Missing Security Headers (HSTS, X-Frame-Options, etc.) Low Open
The following headers are not set: Strict-Transport-Security, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy, Permissions-Policy. Vercel sets HSTS automatically on custom domains, but explicit configuration is better practice.
🔵
Game Integrity: Server-Side RNG ✓ Info ✓ Secure
All game outcomes are computed server-side via gameSpinAction(). The client sends only the bet amount — the server controls all RNG, payout calculation, and balance updates. Provably-fair hash chain is implemented. The wallet uses optimistic deduction client-side but always syncs from the authoritative server balance after each spin. This is the correct pattern for a fair game.
🔵
bcrypt Password Hashing ✓ Info ✓ Secure
Passwords are hashed with bcrypt (cost=12), which is the correct industry standard. The plaintext password is never stored or logged. The compare uses bcrypt.compare() which is timing-safe.
