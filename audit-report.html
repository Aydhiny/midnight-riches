Security Audit
61 / 100
Critical In-Memory Rate Limiter Fails Under Load Balancing
The rate limiter uses a Map<string, RateLimitEntry> stored in Node.js process memory. In any multi-instance deployment (Vercel Serverless, Docker, etc.), each cold-start gets a fresh map. A user can bypass all rate limits by simply hitting a different function instance. This affects signup abuse, spin flooding, checkout spamming, and brute-force protection simultaneously. 
// src/lib/server/rate-limiter.ts — current (broken at scale)
const store = new Map<string, RateLimitEntry>();  // resets on every cold start

// Fix: replace with Upstash Redis or Vercel KV
import { Ratelimit } from "@upstash/ratelimit";
import { Redis } from "@upstash/redis";
const ratelimit = new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(10, "10 s") });
src/lib/server/rate-limiter.ts
Critical Content Security Policy Uses unsafe-eval + unsafe-inline
The CSP header includes 'unsafe-eval' and 'unsafe-inline' for script-src. This completely defeats XSS protection — any injected inline script or eval() call will execute. For a financial application handling real money, this is a significant attack surface. unsafe-eval is often required by PixiJS/WebGL but can be scoped with nonces instead. 
// next.config.ts — current (insecure)
"script-src 'self' 'unsafe-eval' 'unsafe-inline' ..."

// Better: use nonces for known inline scripts, keep unsafe-eval only for WebGL
"script-src 'self' 'nonce-{nonce}' 'unsafe-eval'"  // drop unsafe-inline
next.config.ts → headers()
Critical No Database-Level Audit Trail for Admin Financial Actions
Admin balance adjustments write directly to the wallet table with no immutable record of who changed what, when, by how much, and why. The reason parameter is accepted by adminAdjustBalance but never written to any log table. In a regulated gaming context this is a compliance failure — you cannot reconstruct a user's financial history after a dispute. 
// Missing: admin_audit_log table
const adminAuditLogs = pgTable("admin_audit_logs", {

  targetId:   text("target_id").references(() => users.id),
  action:     text("action").notNull(),         // "balance_adjust", "role_change"
  before:     jsonb("before"),
  after:      jsonb("after"),
  reason:     text("reason"),
  createdAt:  timestamp("created_at").defaultNow().notNull(),
});
src/server/actions/admin.ts · src/lib/db/schema.ts
High allowDangerousEmailAccountLinking Enables Account Takeover
All three OAuth providers (Google, GitHub, Discord) have allowDangerousEmailAccountLinking: true. If an attacker controls an OAuth provider's account with the same email as a victim, they can silently link to and take over that account. This is especially dangerous because credentials users have passwords that become irrelevant after linking. The NextAuth docs name this "dangerous" explicitly.
src/lib/auth/index.ts lines 33, 39, 46
High TOTP 2FA Secret Stored Unencrypted
The twoFactorSecret column stores the raw TOTP seed in plaintext. If the database is breached, an attacker has both the password hash AND the second factor — negating 2FA entirely. Secrets should be encrypted with a server-side key (AES-256-GCM) before storage. The server key lives in env, so a DB-only breach doesn't expose 2FA.
src/lib/db/schema.ts · src/lib/2fa/index.ts
High Referral System Has No Fraud Detection
The referral system gives real credits (100cr referrer + 50cr referred) with no guards against: same IP address creating multiple accounts, bot-generated referral chains, or a user referring themselves via a different email. The referral code pattern (MR-[USER_ID_SLICE]-[4BYTES]) is guessable. A motivated bad actor could manufacture thousands of referral bonuses.
src/server/actions/referrals.ts
Medium Anomaly Detection is Advisory Only — Never Blocks
The anomaly-detector.ts correctly flags statistical outliers (win rate >65%, payout ratio >3×) but explicitly states it does NOT block. Flagged sessions continue unimpeded. This is fine for legitimate users but means cheating is detected and ignored. Flags should at minimum trigger an automatic review queue or temporary session suspension.
src/lib/security/anomaly-detector.ts
Medium Self-Exclusion Only Blocks Spins, Not Purchases
spinGuard.ts correctly checks self-exclusion before every spin. However, a self-excluded user can still purchase credits via Stripe. They simply cannot spend them. This is a responsible gambling gap — exclusion should block the entire gambling loop including deposit/purchase actions.
src/lib/security/spin-guard.ts · src/server/actions/stripe.ts
Medium Spin-Guard lastSpinTime Also In-Memory
Like the rate limiter, lastSpinTime uses a Map<string, number> in process memory. On Vercel, each serverless invocation is potentially a new process. The minimum 500ms spin interval (anti-spam) is therefore unenforceable in production. Move this check to the database using a last_spin_at column on the wallet or game_sessions table.
src/lib/security/spin-guard.ts lines 8–9
Good Auth & Input Validation Are Solid
All server actions use Zod schemas for input validation. JWT strategy with 30-day expiry is appropriate. Server actions use allowedOrigins CSRF protection. The 2FA flow design (error-based signaling from authorize(), separate bypass provider) is clever and correct. Email verification gates are applied consistently across Stripe, payment methods, and referrals.

Performance Audit
55 / 100
Critical Zero Caching Strategy — Every Request Hits the Database
There is no caching layer anywhere in the stack. Admin KPIs, wallet balances, leaderboards, jackpot values, game session histories — every page load makes fresh database queries. For a gaming app where the same data is read hundreds of times per second, this will cause latency spikes and unnecessary Neon compute costs. At minimum, Next.js unstable_cache or React's cache() should wrap frequently read data. Jackpot values and leaderboards are prime candidates. 
// Missing pattern: cache jackpot reads
import { unstable_cache } from "next/cache";
export const getCachedJackpot = unstable_cache(
  async () => db.query.jackpots.findFirst({ ... }),
  ["jackpot"],
  { revalidate: 30 }  // 30 second cache
);
High JWT Callback Queries Database on Every Request
The jwt callback in auth/index.ts fetches role and image from the database every single time a JWT is created or refreshed (lines 113–132). In a session-based app this fires on every authenticated page load. It creates a DB round-trip for the most common operation in the entire application. Role and image should be stored directly in the JWT token and only re-fetched when explicitly updated via trigger === "update".
src/lib/auth/index.ts lines 113–132
High Game Session History Has No Pagination Limit
The anomaly detector queries game sessions without a hard cap. A user with 10,000 sessions will cause a massive memory allocation and slow query on every spin. The history page similarly loads unbounded result sets. All list queries need a reasonable LIMIT plus cursor-based pagination. Drizzle makes this straightforward with .limit(50).offset(page * 50).
src/lib/security/anomaly-detector.ts · src/app/(main)/history/page.tsx
High Logo Image: 1,987 KiB Served at 22×22px
The midnight-riches-logo.png is a 1536×1024 PNG weighing 1.99 MB. It is displayed at 22×22 pixels in the demo slot's "cabinet header". Even with the Next.js <Image> swap now in place, the source file itself is vastly oversized. At 22px display size, the source only needs to be ~44×44px (2× for retina). The 1.99 MB source should be replaced with a properly sized PNG/SVG. Similarly, five-reel-deluxe.png (1.24 MB) and cascading-reels.png (826 KB) are displayed at ~533×300 but are multi-megapixel originals.
public/images/ — all game thumbnails and logo
Medium PixiJS Symbol Textures Not Optimized (250×250 PNG at 76px)
All 12 game symbols (Bell, Cherry, Diamond, etc.) are 250×250 PNG files served as PixiJS textures and also displayed as <img> tags at 76×76px in the paytable. These are loaded raw — PixiJS does not use Next.js image optimization. Each texture download is 11–31 KB when it could be 2–5 KB as a properly sized WebP. A build-time sharp script to generate 100×100 WebP versions for the PixiJS pipeline would cut texture load time by ~70%.
public/images/Bell.png, Cherry.png, … (12 files)
Medium No Wallet Balance Zustand → Server Sync on Navigation
The wallet balance is stored in useWalletStore (Zustand) and displayed in the navbar. After a spin win or purchase, the Zustand store is updated client-side. However, if the user opens a second tab or returns after a session, the store is re-seeded from the server. There's no WebSocket or SSE push for real-time balance updates, meaning displayed balance can be stale in multi-tab scenarios.
src/store/wallet-store.ts
Good Dynamic Imports and Code Splitting Are Used Correctly
SlotMachine is dynamically imported with ssr: false, preventing PixiJS from being included in the server bundle. The game page shows a proper loading skeleton during the dynamic import. Turbopack is enabled for faster development builds. The removal of unused packages (@react-three/fiber, @react-three/drei, three, @stripe/stripe-js) was correctly identified and executed.

Architecture Audit
75 / 100
High Financial Operations Lack Database Transactions
The spin action deducts a bet and credits a win in separate database statements with no wrapping transaction. Under concurrent load (auto-spin + network hiccup), the bet could be deducted twice or the win credited twice. For any financial application this is unacceptable. Drizzle supports transactions — every spin that touches two tables must be wrapped in db.transaction(). 
// Current: two separate writes (race condition risk)
await db.update(wallets).set({ balance: newBalance })...
await db.insert(transactions).values({ ... })...

// Required: atomic transaction
await db.transaction(async (tx) => {
  await tx.update(wallets).set({ balance: newBalance })...;
  await tx.insert(transactions).values({ ... })...;
});
src/server/actions/spin.ts
High game/page.tsx Is a 600+ Line Monolith
The game page file contains: ambient background animation, mouse spotlight, music auto-start, a full live-chat widget (~200 lines), live-wins feed, sidebar buttons, and the navigation guard modal — all in a single file. This makes the file untestable, hard to maintain, and causes the entire page to re-render when any sub-component state changes. The LiveChat component alone should be extracted to its own file.
src/app/(main)/game/page.tsx — 600+ lines
Medium Decimal / Number Type Inconsistency Throughout
Database stores amounts as decimal(12,2) which Drizzle returns as strings. Application code converts with Number() or parseFloat() constantly. This creates floating-point risk for currency calculations (e.g., 0.1 + 0.2 !== 0.3). The correct approach is to work in integer cents throughout, or use a dedicated Decimal library (decimal.js) for all arithmetic.
src/server/actions/spin.ts · stripe.ts · wallet.ts
Medium Server Action Error Responses Are Not Standardized
Different actions return different error shapes. Some return { success: false, error: string, code: string }, others return { success: false, error: string } with no code, and some throw exceptions that become unhandled. A shared ActionResult<T> type enforced consistently would eliminate these divergences and make client-side error handling predictable.
src/server/actions/*.ts — inconsistent patterns
Medium Middleware PROTECTED Array Is a Maintenance Liability
Route protection is a hardcoded array: ["/game", "/wallet", "/history", "/settings", "/shop", "/admin", "/achievements"]. Every new route added to the app requires a developer to remember to update this list — and there's no test or lint rule to enforce it. A better pattern is to invert: protect everything by default, explicitly opt routes out (public allowlist), or use file-based conventions (e.g., a _protected marker).
src/middleware.ts
Good Provably Fair System Is Cryptographically Sound
The game uses HMAC-SHA256 with a server seed, client seed, and nonce. The server seed hash is committed before play and revealed after rotation. Nonce increments prevent replay. This is mathematically equivalent to what established crypto casinos use and is a genuinely impressive addition for a solo project. The RTP-calibrated weighted RNG is implemented correctly across all four game variants.
src/lib/game/provably-fair.ts · src/lib/game/rng.ts
Good Folder Structure and Separation of Concerns Are Strong
The project correctly separates: lib/ (pure utilities), server/actions/ (server-side mutations), components/ (UI), store/ (client state), app/ (routing). Server actions are never imported in shared components that could run client-side. The Drizzle schema is the single source of truth for data shapes. This structure would feel familiar to any Next.js senior developer.

Code Quality Audit
70 / 100
High Test Coverage Is Shallow — Zero Integration or E2E Tests
Unit tests exist for RNG, validators, Stripe config, and the provably-fair algorithm — which is good. However, there are no integration tests that test the full server action pipeline (auth → action → DB → response), and no E2E tests covering the critical paths: sign-up → email verification → purchase → spin → win. The most dangerous bugs — race conditions, auth bypasses, payment failures — are exactly the ones unit tests cannot catch.
Missing: tests/integration/ · tests/e2e/
High Logging Is Inconsistent and Missing in Critical Paths
A logger utility exists but is only used in some server actions. Critical events — failed payment webhook processing, admin role changes, self-exclusion activations, 2FA setup/disable, anomaly flags — are not logged. On Vercel, console output disappears after the function completes. For production, logs need to be shipped to a persistent service (Axiom, Logtail, Datadog) with structured fields (userId, action, metadata) that enable querying after an incident.
src/lib/logger.ts — used inconsistently
Medium Hardcoded Strings in UI Components (i18n Gaps)
Despite a solid next-intl setup, several UI strings remain hardcoded in English: the live chat placeholder ("Say something..."), admin dashboard sidebar labels (partially fixed), password strength labels in signup ("At least 8 characters", etc.), and bot chat messages in the game page. The i18n coverage is ~75% — good but not complete.
src/app/(main)/game/page.tsx · src/app/auth/signup/page.tsx
Medium No Environment Variable Validation at Startup
Missing required env vars (Stripe key, EmailJS credentials, DB URL) silently fail at runtime, often with cryptic errors deep in a user flow. There is no startup validation using Zod or @t3-oss/env-nextjs. A missing STRIPE_SECRET_KEY should crash the build, not cause a 500 error when a user tries to purchase. 
// Recommended: src/env.ts
import { createEnv } from "@t3-oss/env-nextjs";
export const env = createEnv({
  server: {
    DATABASE_URL:         z.string().url(),
    STRIPE_SECRET_KEY:    z.string().startsWith("sk_"),
    EMAILJS_SERVICE_ID:   z.string().min(1),
    AUTH_SECRET:          z.string().min(32),
  },
  runtimeEnv: process.env,
});
Medium No Error Boundary in the Game Renderer
The PixiJS slot machine renders inside a dynamically imported component with no React Error Boundary wrapping it. A PixiJS crash (WebGL context lost, texture load failure) would unmount the entire game page with a blank white screen. An error boundary would catch this and show a "Game failed to load — reload" message with a recovery button, keeping users in the app.
src/app/(main)/game/page.tsx — SlotMachine dynamic import
Low README and Developer Documentation Are Missing
There is no README.md with setup instructions, env var documentation, or architecture overview. A new developer joining the project has no guidance on how to run it locally, what Stripe webhooks to configure, how to run migrations, or what the game engine RTP targets are. This is a maintainability risk as the project grows.
Good TypeScript Strict Mode + Zod Validation Is Excellent
The project uses TypeScript in strict mode with no significant type suppressions (any casts are rare and commented). All server action inputs are validated with Zod schemas before reaching business logic. The Drizzle schema serves as the runtime type source, eliminating an entire class of "schema drift" bugs common in other ORMs. This is senior-level TypeScript hygiene.

Game Integrity & Responsible Gambling
68 / 100
High Gambling Limits Are Stored But Not Fully Enforced
The responsible gambling UI lets users set deposit limits, loss limits, and session limits. The spin guard checks loss limits. However: deposit limits are never checked against Stripe checkout amounts, session time limits have no enforcement mechanism (no session start timestamp used), and the UI-set limits can be changed without a cooling-off period — a user can disable a self-imposed limit immediately.
src/server/actions/responsible-gambling.ts · src/lib/security/spin-guard.ts
Medium No RTP Transparency Displayed During Gameplay
The game engine is configured with specific RTP values (Classic Fruits 94%, Five Reel Deluxe 95.2%, etc.) but these are not disclosed to users during gameplay. The game selector shows RTPs — which is good. But responsible gaming regulations in most jurisdictions require this information to be persistently visible, not only in a selection screen.
Medium Integrity Validator Runs After Game Is Recorded
The integrity-validator.ts validates spin results after writing to the database. An integrity failure is logged but the transaction has already been committed. The validator should run before the wallet update and reject the spin atomically if validation fails.
src/lib/security/integrity-validator.ts
Good Provably Fair + Anti-Cheat Architecture Is Impressive
Having a provably fair cryptographic system, anomaly detection, spin guard with parallel-spin prevention, and responsible gambling controls in a solo project is genuinely impressive. Most indie gaming projects ship with none of these. The HMAC-SHA256 proof system is mathematically equivalent to what commercial platforms use. This is the project's biggest technical strength.

Honest Verdict

This is a legitimately impressive solo project that would rank in the top 5% of personal portfolio work. The game engineering alone — four distinct RTP-calibrated engines, a provably fair system, spring-physics PixiJS renderer, and cascading/megaways mechanics — is the kind of output most junior-to-mid developers couldn't produce. The decision to add 2FA, email verification gating, responsible gambling controls, and anomaly detection shows genuine security awareness that is rare outside professional teams.

However, this is not production-ready for real money. The in-memory rate limiter, lack of database transactions around financial operations, missing admin audit trail, and unencrypted 2FA secrets are not cosmetic issues — they are the kind of failures that cause real financial harm and regulatory liability. A regulated gaming platform would fail licensing on items #1–4 in the priority matrix alone.

The path forward is clear and achievable. The critical items are 4 targeted fixes, not architectural rewrites. Items #1 (Redis rate limiting) and #2 (db.transaction()) are each under 2 hours of work. With those four done, this project moves from a 64 / C+ to approximately 79 / B+ — comfortably production-ready for a demo or MVP with real but capped stakes.
